Day 1 – AWS Global Infrastructure

Regions

  • Each Region = geographical area with 2+ AZs.
  • Regions are isolated; replication only when configured.
  • Region choice affects latency, cost, compliance, and availability.
  • Not all services exist in every Region → check AWS Regional Services List.

Availability Zones (AZs)

  • Each AZ = 1+ physically separate data centers.
  • Low-latency links (1–2 ms) between AZs.
  • Use multi-AZ for high availability (EC2, RDS, ELB).

Edge Locations

  • Used by CloudFront, Route 53, Global Accelerator.
  • Provide caching, DNS routing, traffic acceleration.
  • Not compute — don’t confuse with AZs.

Local & Wavelength Zones

  • Local Zones → bring AWS closer to metro users.
  • Wavelength Zones → 5G / telecom ultra-low latency apps.
Exam Tip: Which option gives compute near users with single-digit ms latency? → Local/Wavelength Zone.

Fault Tolerance

  • Multi-AZ → survive single AZ failure.
  • Multi-Region → disaster recovery.
  • S3 & DynamoDB → multi-AZ by default; EC2/RDS/EBS → single-AZ unless configured.

Networking & Latency

  • Cross-AZ traffic = free; Cross-Region = billed.
  • Keep data near users to minimize latency and cost.

Global Services

  • Global: IAM, Route 53, CloudFront, WAF, Organizations.
  • IAM = global, no Region selection.
Exam Tip: Is IAM regional or global? → Global ✅
