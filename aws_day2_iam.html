Day 2 – IAM, Users, Roles

IAM Overview

  • AWS Identity and Access Management (IAM) allows secure access control to AWS services.
  • IAM is global — no region selection needed.
  • Manage users, groups, roles, and policies.
  • Use least privilege principle — minimum permissions only.
  • Enable MFA for root and privileged accounts.
  • Policies are JSON-based and define allowed/denied actions.

Users & Groups

  • Create IAM users for individuals; avoid sharing root credentials.
  • Groups manage permissions collectively.
  • Attach policies to users or groups for controlled access.
  • Managed policies = reusable; inline policies = specific to a user/group.
  • Always prefer groups over direct user permissions.

Roles

  • Roles provide temporary permissions that can be assumed.
  • Use cases: cross-account access, EC2/S3 access, Lambda execution.
  • AssumeRole API allows temporary permission granting.
  • Policies attached to roles define what they can do.

Policies

  • Policies define permissions in JSON format.
  • Types: managed policies (AWS or custom) and inline policies (specific to user/group/role).
  • Evaluation logic: Default deny → explicit allow → explicit deny overrides all.

Access Keys & MFA

  • Access keys allow programmatic access via CLI or SDK.
  • Enable MFA for extra security.
  • Root account: avoid daily use; create admin users with MFA.

Exam Tips

  • IAM is global → exam may ask which service is global.
  • Roles are temporary/assumable; users are persistent.
  • Always follow least privilege principle.
  • Cross-account access → use roles, not shared credentials.
  • Protect root account with MFA; limited usage only.
  • Policy evaluation order: Deny > Allow > Default Deny.
